# Biometric Unlock for KeePass Vaults (Android)

Implementation plan for unlocking `.kdbx` vaults with the device fingerprint
sensor (or any Android `BIOMETRIC_STRONG` authenticator) in the Tauri mobile
build. Android only — iOS is out of scope until we scaffold `gen/ios/`.

## Goal

The user opens a recent vault on their phone, taps "Unlock with fingerprint",
verifies, and the existing `Database::open` flow runs with the cached master
password. No password typing on mobile.

Behaviour parity target: KeePassDX. Re-enrolling a fingerprint on the device
must invalidate the cached password (forcing a re-enrollment with the master
password); uninstalling the app must purge the secret.

## Why not just use `tauri-plugin-biometric`

`tauri-plugin-biometric` only exposes `authenticate(reason, opts)` and
`checkStatus()`. It does **not** wrap a `CryptoObject`, so the OS never
authorises a key release inside the prompt. If we used it alone, our only
option would be to store the master password encrypted with a key that lives
in app-private storage (extending the existing `secret_store.rs` AES-GCM
fallback). That keeps the password reachable to anyone who can read app data
on a rooted device or via `adb backup`, regardless of whether the biometric
prompt succeeded — the prompt is a UX gate, not a real protection. KeePassDX
explicitly avoids that. We do too.

The right primitive is `BiometricPrompt.authenticate(prompt, CryptoObject)`:
the Keystore key has `setUserAuthenticationRequired(true)` so it cannot be
used at all without a fresh biometric authorisation, and the cipher is only
returned inside the success callback. This is a one-screen plugin we write
ourselves.

## Architecture

```
crates/uncloud-web (Dioxus, WASM)
   └── hooks/biometric.rs            ← thin Tauri-invoke wrapper, no-op on web
       │
       │  invoke("plugin:uncloud-biometric|...")
       ▼
vendor/tauri-plugin-uncloud-biometric/
   ├── src/lib.rs                    ← Rust plugin shim, registers handlers
   ├── android/
   │   └── src/main/java/.../        ← Kotlin: BiometricPrompt + Keystore +
   │                                   SharedPreferences for ciphertext
   └── permissions/                  ← capability allowlist
```

The plugin is vendored, same as `tauri-plugin-native-audio`, because it
contains custom Kotlin we own. Outside Android the plugin's commands return
`Err("not supported on this platform")`; the web/desktop builds use a no-op
hook that reports `available=false`.

### Plugin command surface (JS-facing)

| Command                         | Args                              | Returns                              |
|---------------------------------|-----------------------------------|--------------------------------------|
| `status()`                      | —                                 | `{ available, reason? }`             |
| `is_enrolled({ user_id, vault_id })`        | both: `String`               | `bool`                               |
| `enroll({ user_id, vault_id, secret })`     | `secret: String` (master pw) | `Ok(())` or `{ error }`              |
| `unlock({ user_id, vault_id })`             | both: `String`               | `{ secret }` or `{ error }`          |
| `clear({ user_id, vault_id })`              | both: `String`               | `Ok(())`                             |

`vault_id` is the server-side `FileResponse.id` — already stable per file and
already on the unlock screen. `user_id` is the current Uncloud account id from
`AuthState`. The two together form the storage key (see below); the plugin
treats them both as opaque strings.

### Android storage layout

- **Keystore alias**: `uncloud.vault.{user_id}.{vault_id}` — AES-256,
  `BLOCK_MODE_GCM`, `ENCRYPTION_PADDING_NONE`,
  `setUserAuthenticationRequired(true)`,
  `setInvalidatedByBiometricEnrollment(true)`. No timeout — every unlock
  triggers a prompt. Keying by `{user_id}.{vault_id}` lets the same vault
  file be enrolled separately for each Uncloud account on the device, and
  ensures user A's enrolment can't unlock the same kdbx for user B.
  `user_id` comes from `AuthState` and is passed in by the caller; the
  plugin treats it as opaque.
- **SharedPreferences**: file `uncloud_biometric_vaults`, key
  `{user_id}.{vault_id}`, value `base64(iv || ciphertext)`. The IV must be
  persisted because GCM decryption needs the same IV that was used at enrol
  time, and that IV is generated by the Keystore itself on the encrypt cipher.
- Uninstall removes both — no extra cleanup needed.

### Enrol flow (Kotlin, sketch)

1. Generate Keystore key for alias `uncloud.vault.{vault_id}` (overwriting
   any stale alias).
2. `Cipher.getInstance("AES/GCM/NoPadding")`, `cipher.init(ENCRYPT_MODE, key)`.
3. `BiometricPrompt.authenticate(prompt, CryptoObject(cipher))` with
   `setAllowedAuthenticators(BIOMETRIC_STRONG)`.
4. On success: `result.cryptoObject.cipher.doFinal(secret.toByteArray())`,
   prepend the cipher's IV, base64-encode, write to SharedPreferences.

### Unlock flow

1. Read `iv || ciphertext` from SharedPreferences.
2. Build `Cipher.getInstance("AES/GCM/NoPadding")`,
   `cipher.init(DECRYPT_MODE, key, GCMParameterSpec(128, iv))`.
3. `BiometricPrompt.authenticate(prompt, CryptoObject(cipher))`.
4. On success: `result.cryptoObject.cipher.doFinal(ciphertext)` → return
   the recovered master password.
5. On `KeyPermanentlyInvalidatedException` (re-enrol of fingerprints) or
   `InvalidKeyException`: clear the SharedPreferences entry and return
   `{ error: "biometric_invalidated" }` — UI prompts the user to re-enrol.

## Frontend integration

- New module `crates/uncloud-web/src/hooks/biometric.rs` exposes
  `status()`, `is_enrolled(vault_id)`, `enroll(vault_id, secret)`,
  `unlock(vault_id)`, `clear(vault_id)`. On non-Tauri builds and on
  desktop Tauri it short-circuits to `available=false`.
- `VaultUnlockCard` (`components/passwords.rs`):
  - On mount, call `is_enrolled(user_id, file_id)`. If true, automatically
    fire `unlock(...)` — the user opened the vault, the prompt is the next
    expected step. Don't gate it behind a button tap. The same screen
    keeps the password field visible as a fallback for "Cancel" /
    `KeyPermanentlyInvalidatedException`.
  - After a successful **manual** unlock, if `status().available && !is_enrolled`,
    show a one-time prompt: "Enable biometric unlock for this vault?"
    Yes → call `enroll(user_id, file_id, master_pw)` (the password is
    already in the component's signal at that point). No / Not now → set a
    "skipped" flag in `localStorage` keyed by `{user_id}.{file_id}` so we
    don't ask again until they open the vault settings.
  - "Forget biometric for this vault" action in the recents-row dropdown
    calls `clear(user_id, file_id)` and removes the local "skipped" flag too.

#### Idle auto-lock

Promote the unlocked-vault `Signal<Option<VaultState>>` from the
`PasswordsView` component to an app-level context (alongside `AuthState` /
`ThemeState`). Keep a `last_active_at: DateTime<Local>` in `VaultState`,
bumped on any vault interaction (typing in search, opening an entry,
navigation back to the route). A `use_future` ticker checks every 30 s and
clears the vault if `now - last_active_at > 5 min`. Result: navigating
away and back inside the window is free; idle past the window forces a
fresh biometric prompt (or master-password entry on devices without
biometric). 5 minutes is the default; not exposing it as a setting yet.
- Recents UI: small fingerprint icon next to vault entries that have
  biometric enrolled (cheap visual cue; queried once per `recents` list
  refresh).

No server-side changes. Biometric state is intentionally device-local —
enrolment on phone A must not unlock the vault on phone B.

## Pieces of work, in dependency order

1. **Vendor the plugin scaffold** — `vendor/tauri-plugin-uncloud-biometric/`.
   Copy structure from `vendor/tauri-plugin-native-audio/` but minimal: only
   Android folder, no iOS/desktop. Wire into `crates/uncloud-desktop/Cargo.toml`
   the same way native-audio is wired (path dependency, gated by
   `cfg(target_os = "android")`).
2. **Kotlin plugin** — `BiometricPlugin.kt`: implement the five `@Command`s
   above. Keystore + BiometricPrompt + SharedPreferences. ~250 lines.
   Manifest additions: `<uses-permission android:name="android.permission.USE_BIOMETRIC" />`
   in the desktop crate's `gen/android/app/src/main/AndroidManifest.xml`.
3. **Rust shim** — `vendor/.../src/lib.rs`: `PluginBuilder::new("uncloud-biometric")`,
   `.invoke_handler(...)` for each command, JSON arg/return types shared
   with the JS side. Stub `cfg(not(target_os = "android"))` returns an
   "unsupported" error from every command.
4. **Capabilities** — add `uncloud-biometric:default` to the desktop
   crate's `capabilities/default.json` so the JS side can call it.
5. **Web hook** — `hooks/biometric.rs` with the Tauri-invoke wrappers, and
   a `cfg`-style guard so non-Tauri builds compile.
6. **Vault UI integration** — modify `VaultUnlockCard` and the recents
   dropdown in `components/passwords.rs`. Add the "Enable biometric unlock?"
   prompt after a successful manual unlock.
7. **Manual test on a real Pixel / mid-range device** — emulators have
   working biometric (`adb -e emu finger touch 1`) but the
   `KeyPermanentlyInvalidatedException` path is hard to reproduce there;
   the real-device check is for the re-enrol invalidation behaviour.

## Build / CI

- `tauri-plugin-uncloud-biometric` is vendored, so `cargo check -p uncloud-desktop`
  picks it up via path dependency. No new external download.
- Android build (`pnpm tauri android build` or whatever we end up using)
  needs `androidx.biometric:biometric` in the plugin's `build.gradle.kts`.
  Pin the version, document it.
- The current CI matrix only builds Linux RPM/DEB and Windows NSIS; Android
  build is not on CI yet. This work doesn't change that — it just adds a
  feature that *requires* an Android build to test. Adding an Android job to
  `release-desktop.yml` is a separate piece of work, tracked elsewhere.

## Resolved decisions

- **Enrolment flow**: auto-prompt on first manual unlock of a vault (one-time,
  per `{user_id, vault_id}`, with a "skipped" flag in `localStorage` to
  avoid re-asking).
- **Multi-account keying**: storage keyed by `{user_id}.{vault_id}` so the
  same kdbx file can be enrolled separately for each Uncloud account on
  the device.
- **Idle TTL**: 5 minutes. Vault state lifted to app-level context with a
  `last_active_at` timestamp and a 30 s ticker that clears it past the
  window. See "Idle auto-lock" above.
